## The Constructive build surface (what you actually assemble)

A Constructive app is built from **three distinct surfaces** â€” keep them straight, because they use
different tools and the "flows"/"Blocks" vocabulary covers **only the first**:

1. **AUTH surface â€” pick the flows your app needs.** The **18 auth "flows"** (and the ~68 "Blocks"
   they install) are the **auth / account / org / shell** experience: sign-in/up, password/MFA/step-up,
   account settings, organization + member/invite management, the authed shell. Pick from the **flow
   catalog** ([flow-catalog.md](./flow-catalog.md) / [flows.json](./flows.json) / the constructive-blocks
   skill); each flow maps to a set of blocks you `shadcn add`. **Flows are auth-only â€” they are NOT
   general "app flows" and contain none of your domain UI.** (Speedrun: S5.)
2. **DATA surface â€” model your tables (the blueprint).** Your domain lives in a declarative
   **blueprint** `definition` (tables + policies + relations) the platform provisions. Author it from
   the **brief** and generate it: **`node scripts/scaffold-provision.mjs build/app-brief.yaml <app>`**.
   (The brief path is whatever you author; under the **per-app state convention** it canonically lives at
   `build/<app-id>/app-brief.yaml` â€” app-id = the brief's `naming.db_name` â€” and `build/app-brief.yaml`
   is the legacy single-app default. See [brief-grammar.md](./brief-grammar.md) + Rule 7.) (Speedrun: S2.)
3. **DOMAIN UI surface â€” build entity UI FROM the data model.** Every business entity's CRUD UI comes
   from a **separate** system: the **`constructive-frontend`** CRUD Stack + runtime-generic `_meta`
   meta-forms (`DynamicFormCard` / `useMeta` / `DynamicField`), generated by
   **`node scripts/scaffold-frontend.mjs build/app-brief.yaml <app>`**. (Speedrun: S7.)

> **Flows are auth only; everything domain-specific is (2) + (3).** Do **not** go looking for a
> "todos flow" or a "create-record block" â€” there is none. The flow catalog never points to your
> entity UI on purpose: that is what the data model (2) and the CRUD body (3) are for.

**Conventions used below:**
- `<app>` (the **scripted CLI arg**) = the **WORKSPACE ROOT** (the dir holding `packages/`, e.g.
  `myapp` / `.`) â€” the **single, unified** argument all three scaffolders take:
  `scripts/scaffold-provision.mjs`, `scripts/scaffold-frontend.mjs`, **and** `scripts/wire-app.mjs
  --app`. Each one DERIVES the app package (`packages/app`, or a root-level `app/`) itself, so you pass
  the **same** `<app>` to all three. (`wire-app` still accepts an explicit app dir for back-compat;
  given a workspace root with **no** app package under it, it FAILS LOUDLY â€” non-zero, naming the root â€”
  rather than silently retargeting a stale leftover from another build.) The pgpm template nests the app
  under `packages/app/` (its `package.json`/`src/`/`next.config.ts` live there).
  **Heads-up on a second usage:** in the **manual-fallback / hand-run** blocks below (the "do it by hand
  if a script fails" prose, and the raw `pnpm codegen` / `next build` / `shadcn add` commands), `<app>`
  instead denotes that **app package dir** â€” i.e. paths like `<app>/.env`, `<app>/src/...`, and `cd
  <app>` mean **`packages/app`**, because those commands run **inside** the app package. So: pass the
  **workspace root** to the three scaffolder CLIs; read `<app>/â€¦` as **`packages/app/â€¦`** in the
  hand-run/manual blocks. Each such block also says so locally.
- `<sub>` = the subdomain = **the database name with underscores** (db `my_app` â†’ `my_app`). Per-DB
  endpoints: `auth-<sub>` (auth), `admin-<sub>` (orgs/members), **`api-<sub>` (your DATA â€” NOT
  `app-public-<sub>`)**. The HTTP **`Host`** header drives routing (a URL alone still 404s).
- Provisioning/blueprint writes go to **`modules.localhost:3000`** (they 404 on `api.localhost`).
- Pick a free port pair up front, e.g. app `3081` + local registry `4081` (avoid clashes with other runs).

> **Where the backend lives â€” `constructive.config.json` (single config source).** The infrastructure
> coordinates every script above uses come from one file at THIS SKILL's root,
> **`constructive.config.json`**: the **hub** the app reads/writes (host, port, GraphQL path, the
> role-subdomain host pattern), the **hub database** (the physical Postgres DB that holds every app's
> schema), the component **registry** URL/port, and the app's **dev-server port base**. The defaults are
> the standard local coordinates shown in these conventions, so on a normal local setup you change nothing.
> **To point a build at a different backend or different ports, do NOT edit the scripts** â€” every value is
> overridable by a `CONSTRUCTIVE_*` environment variable (e.g. `CONSTRUCTIVE_HUB_HOST`,
> `CONSTRUCTIVE_HUB_PORT`, `CONSTRUCTIVE_HUB_DATABASE`, `CONSTRUCTIVE_REGISTRY_BASE_URL`,
> `CONSTRUCTIVE_APP_DEV_PORT_BASE`; the exact var for each key is in the file's `env` maps). Read a
> resolved value from the shell with `node scripts/lib/config.mjs get <dotted.key>` (e.g. `hub.port`), or
> relocate the whole file with `CONSTRUCTIVE_CONFIG_PATH`. One file, env-overridable, so the same scripts
> target any environment. Full detail: [secrets-and-config.md](./secrets-and-config.md).

## S2 â€” Provision the DB (GENERATE the 6 files from the brief; OBJECT-FORM grants; normalized membership SQL)

**Generate the provision package from the brief (the default path):**

```bash
# Emits all 6 provision files into <app>/packages/provision/src/ (config/helpers/blueprint/
# create-db/provision.ts + schemas/core.ts) FROM build/app-brief.yaml: per table it resolves the
# brief's policy INTENT (owner | org-membership | member-owner | public-read+owner-write |
# public-lookup) â†’ nodes[] + object-form grants + policies[], computes the create-db module list
# from the chosen flows (NEVER ['all']), and emits the auth-preset appendix. <app> = the WORKSPACE
# ROOT (the dir holding packages/, e.g. `myapp` / `.`). config.ts/helpers.ts/blueprint.ts are copied
# verbatim; only schemas/core.ts is brief-derived. Re-run after any brief edit (idempotent).
# It ALSO re-stamps packages/provision/package.json with the runtime deps (tsx, @constructive-io/sdk,
# pg, dotenv, node-type-registry) AND self-installs them (pnpm install --prefer-offline at <app>) so
# `pnpm run create-db` is HANDS-FREE â€” no separate `pnpm install` step before it. (Self-install is a
# warm-store no-op on a re-run, and is skipped when <app> isn't yet an initialized pnpm workspace.)
# The brief path is a positional â€” pass build/<app-id>/app-brief.yaml under the per-app state
# convention (app-id = naming.db_name), or build/app-brief.yaml for the single-app default.
node scripts/scaffold-provision.mjs build/app-brief.yaml <app>
```

Then run it (same as below): `cd packages/provision && pnpm run create-db && pnpm run provision`,
and VERIFY the grant landed (step 3 below â€” the platform grants `authenticated` natively, so this is a
read-only check). No `pnpm install` is needed first â€” scaffold-provision already installed the provision
deps (above).

Everything from here down in S2 is the **method the generator automates + your hand-edit fallback**:
the same object-form-grant / default-owner-policy / module-list / membership-SQL rules the generator
already bakes in. Skim it so you can read the generated `schemas/core.ts`; only hand-edit when the
generator can't express your shape (use the `nodes_raw`/`policies_raw` brief escape-hatches first â€”
[brief-grammar.md](./brief-grammar.md)).

### Manual fallback (if the generator can't express a shape)

1. Copy the **6 provision files verbatim** from
   **[skill-supplements.md](./skill-supplements.md) "Phase 2.3: Blueprint Provision Template"** into
   `packages/provision/src/` (`config.ts`, `helpers.ts`, `blueprint.ts`, `create-db.ts`, `provision.ts`,
   `schemas/core.ts`). Set your tables in `schemas/core.ts`.
   - **Grants MUST be object-form** (already in the template): `grants: [{ roles: ['authenticated'],
     privileges: [['select','*'],['insert','*'],['update','*'],['delete','*']] }]`. **Never** the stale
     `grant_roles + bare-tuple` form â€” it lands no GRANT and every write 403s (the #1 prior workaround).
   - **DEFAULT POLICY = owner-scoped `AuthzDirectOwner`** (the template's `schemas/core.ts` ships this).
     `nodes: ['DataId', 'DataDirectOwner', â€¦]` (DataId gives the `id` PK for update/delete; F18) +
     `use_rls: true` + `policies: [{ $type: 'AuthzDirectOwner', privileges:[â€¦], permissive:true,
     data: { entity_field: 'owner_id' } }]` (key is `entity_field`, value = the owner column). The client
     sets `ownerId` to the authed user's id on create (WITH CHECK blocks any other value = anti-spoof).
     đźš¨ **Do NOT use `AuthzEntityMembership` + `membership_type: 2` on a basic app** â€” with no org modules
     it FAILS HARD (`NOT_FOUND (memberships_module)`) and the table is never created (RLS-POLICY-001).
     App-wide shared pool instead? `AuthzAllowAll` (no ownership). Org tenancy needs the `b2b` preset.
   - **FK prereq:** `owner_id` FKs to the per-tenant users table, so provision the authed user via the
     TENANT auth endpoint (`auth-<sub>.localhost` signUp), not base `auth.localhost`, or insert FK-violates.
   - `create-db.ts` ships the explicit **`auth:email`** module list (NEVER `['all']`; scoped modules are
     `['name',{scope}]` tuples, NOT colon strings â€” PROVISION-001). For OAuth use `auth:sso`; any org flow â†’ `b2b`.
   - `provision.ts` membership/email SQL already uses the **dash-collapse-tolerant** match (`_`â†’`%`
     wildcard) so it fires on the live hub's dash-collapsed schema names. Don't revert it to `LIKE '%<db>%'`.
   - `provision.ts` also ships the **users-table self-update control-plane step** (a
     `createSecureTableProvision` self_update policy on the modules endpoint) â€” REQUIRED for `updateUser`
     to persist; without it updateUser is a silent 200-but-0-rows no-op (RLS-USERS-UPDATE-001).

2. Run it, capturing the subdomain:
```bash
cd packages/provision
pnpm run create-db          # signs up + provisions + writes ../../.env (DATABASE_ID/NAME, ACCESS_TOKEN, OWNER_ID)
pnpm run provision          # creates tables/RLS + object-form grants + approves membership + verifies email + users self-update policy
```

3. **VERIFY the grant landed.** As the `authenticated` role you must be able to insert *and* read back.
   The platform grants `authenticated` its 4 privileges on the app schema AND the users-table
   `auth_upd_self_update` policy **natively** (PLATFORM-GAPS.md GAP-1a, CLOSED), so this is normally a
   read-only confirmation â€” exactly what the Phase 2.2 gate asserts (the 4 privileges + the self_update
   policy). Only on a deployment that **predates** that fix do you apply it by hand
   (`HUBDB="${PG_HUB_DATABASE:-constructive}"`, find `<APPSCHEMA>`/`<USERSSCHEMA>` via `SELECT schema_name
   FROM information_schema.schemata WHERE schema_name LIKE '%app-public'` / `'%users-public'`):
```bash
psql "$HUBDB" -c "GRANT USAGE ON SCHEMA \"<APPSCHEMA>\" TO authenticated;
                  GRANT SELECT,INSERT,UPDATE,DELETE ON ALL TABLES IN SCHEMA \"<APPSCHEMA>\" TO authenticated;"
# users self-edit (updateUser) â€” emit auth_upd_self_update only if the platform didn't (RLS-USERS-UPDATE-001):
psql "$HUBDB" -c "GRANT UPDATE ON \"<USERSSCHEMA>\".users TO authenticated;
                  CREATE POLICY auth_upd_self_update ON \"<USERSSCHEMA>\".users FOR UPDATE TO authenticated USING (id = jwt_public.current_user_id());"
```

3b. **B2B ONLY â€” org/membership state is now PLATFORM-NATIVE (no reconcile step).** On the b2b tier
    (`modules.preset: b2b` â€” the chosen flow's `backend.modules` carry the org-scoped `memberships`/`hierarchy`
    tuples: `organization`/`org-members`/`org-roles`/`org-invites`/`app-memberships`) the platform grants the
    org tables + `create_entity` bit and **self-seeds** the personal-org `org_memberships_sprt` row every
    `AuthzEntityMembership` table's RLS reads, on signup (PLATFORM-GAPS.md GAP-1b/1c, CLOSED 2026-06-15). So a
    fresh email-password signup's `createCompany(entityId = their user id)` / org-member writes persist
    immediately â€” no reconcile. Verify: a fresh signup's `createCompany(entityId = their user id)` succeeds
    end-to-end (row visible on reload). (Minting a brand-NEW org via `createUser(type=2)` is still
    operator-only â€” PLATFORM-GAPS.md GAP-6, OPEN.)

## Checkpoint + run-state after every green gate (Rule 7)

After **each** phase's `./scripts/verify-phase.sh` gate passes (and **before** starting the next phase):

1. **Update the run-state** â€” set this phase's fields true / fill in the strings (subdomain,
   endpoints, package names). Start from `fixtures/run-state.template.json` if the file does not exist yet.
   `verify-phase.sh` **reads** this file; it never writes it. **Where it lives (per-app state
   convention):** when you export `APP_ID=<app-id>` (the app-id is the brief's `naming.db_name` â€” already
   plain lowercase; the resolver also strips any non-`[a-z0-9]` char defensively), the gates read
   **`build/<app-id>/run-state.json`** + **`build/<app-id>/app-brief.yaml`**, so concurrent apps each keep
   their own state. With `APP_ID` **unset** the readers collapse to the **legacy singleton**
   `build/run-state.json` + `build/app-brief.yaml` â€” the default for the single-app golden path.
   (`verify-phase.sh` even auto-derives `APP_ID` from a resolved spec's `db_name`; explicit `--spec PATH` /
   `--state PATH` always win. There is no `--app` flag â€” set `APP_ID` in the env.)
2. **Git-commit (or tag) the green state** so you can roll back to the last passing phase:
   ```bash
   git add -A && git commit -m "chore(build): phase <N> green (verify-phase.sh <gate> passed)"
   git tag green-phase-<N>      # optional but makes rollback one command
   ```
   Record the tag/sha in `run-state.json` â†’ `checkpoints.phase_<N>`.

**Rollback (when a later phase wedges the tree):** reset to the last green checkpoint and re-provision â€”
do not hand-unpick a broken state:
```bash
git reset --hard green-phase-<last-green>   # or the recorded sha
# then re-run that phase's provision/codegen step and its verify gate
```
