# Phase 4 — Branch A: Blocks Frontend On-Ramp

**Goal:** Make the scaffolded Next.js app able to install and run **Constructive Blocks** — copy-in
React blocks (sign-in card, account, membership/invite flows) distributed via a shadcn registry
(`@constructive/<block>`) that bind to the host app's per-application generated GraphQL SDK.

**When to run this:** this is **Phase 4, branch A** (the conditional first part of the mainline
**UI / Blocks** phase) — run it after **Phase 3 (Frontend + SDK)** and before the Phase 4 CRUD body,
**only if** the app brief asks for auth/account/org/membership UI you would otherwise hand-build.
Blocks replace hand-authored auth and account flows; they do **not** replace your business-entity CRUD
(that is the Phase 4 CRUD body). If the brief needs none of these flows, skip branch A and go straight
to the CRUD body.

> ### What blocks (and "flows") ARE / are NOT
> **Blocks + the 18 auth "flows" cover AUTH, ACCOUNT, ORG, and SHELL only** — sign-in/sign-up cards,
> password/MFA/step-up, account settings, organization + member/invite management, and the app shell
> they live in. That is their *entire* scope.
>
> **They are NOT general "app flows" and NOT your domain entity UI.** Your business entities — the
> tables in the brief's `data_model` (e.g. `posts`, `contacts`, `invoices`) — get their UI from a
> **separate** system: the **CRUD body** (Phase 4), built from the **`constructive-frontend`** skill's
> runtime-generic CRUD Stack + `_meta` meta-forms (`DynamicFormCard` / `useMeta` / `DynamicField`) and
> generated by **`node scripts/scaffold-frontend.mjs build/app-brief.yaml <app>`**. Do **not** look for
> a "todos block" or a "create-record flow" — there isn't one, and shouldn't be. Pick blocks/flows for
> the *auth surface*; build entity UI from the *data model*.

> **Scope:** this reference is the **self-contained** Blocks playbook for the builder — it adapts the
> generic install/wire/check/author flow to **this template**, where the SDK already exists at
> `src/graphql/sdk/{auth,admin}` and a provider is already wired. Everything you need (the preflight
> `check-sdk.mjs`, the fallback registry serve, the manifests) is bundled in this skill's `scripts/`; you
> do **not** need a separate `constructive-blocks` skill checked out beside this one.

## Base vs b2b tier (what the template ships, and what is opt-in)

The scaffold is **tiered**, keyed off your chosen flow's `flows.json` `backend.modules`:

- **BASE (default) = the `auth:email` scaffold.** Ships the auth + account contract below and **no org/b2b
  code at all** — no org pages, no org admin hooks, no members UI, no compat shim. A base app codegens +
  builds clean with **zero dangling org imports**; you never stub a missing `@sdk/admin` org hook.
- **B2B = opt-in** (the registry ORG BLOCKS + provisioning the org modules). Detection + the exact install
  set are in Step 4.0 below — that is where you read the flow and pick the tier.

## What is already true after Phase 2.6 (do not redo)

The base sandbox template (`nextjs/constructive-app`, auth:email tier) gives you most of the blocks contract
for free:

| Blocks needs… | Template already has it | Evidence |
|---|---|---|
| A generated **auth** + **admin** SDK exporting the ORM layer (`configure`, `OrmClientConfig`, `GraphQLAdapter`, `QueryResult`, `GraphQLError`) | `src/graphql/sdk/{auth,admin}` written by `pnpm codegen` (Phase 2.6) | `graphql-codegen.config.ts` outputs `./src/graphql/sdk/{admin,auth,app}` |
| One shared `QueryClientProvider` | `AppProvider` mounts it in `src/components/app-provider.tsx` | `<QueryClientProvider client={queryClient}>` |
| A portal root for overlays (dialogs/menus) | `AppPortalRoot` in `src/app/layout.tsx` | `src/components/portal-root.tsx` |
| A per-request bearer token source | `TokenManager.getToken('admin').token?.accessToken` | `src/lib/auth/token-manager.ts` |

**Blocks therefore need only `auth` + `admin`** — both already generated. There is **no second codegen**
and **no `--orm` flag change**: the template's generated SDK already exports the ORM layer (proven by a
clean `tsc` of `BlocksRuntime` + a block against it). What is missing is purely **binding + host deps +
provider wiring**, which the six steps below add.

> 🗂️ **Layout — where to run these steps.** `pgpm init` unpacks the app into a **nested `packages/app/`**
> (package `constructive-boilerplate`): its `src/`, `tsconfig.json`, `next.config.ts`, `components.json`,
> `globals.css`, `.env*` all live **inside `packages/app/`**, beside the `packages/provision` sibling; the
> pnpm WORKSPACE ROOT is the parent. Below, **`<app>` = that app dir (`packages/app`)** — run every per-app
> command (tsconfig/next.config/globals.css edits, `pnpm add`, `shadcn add`, codegen) **in `<app>`**. (An
> older scaffold put the app at the repo root; `node scripts/wire-app.mjs` and the toolkit gates auto-detect
> either, so paths below say `<app>` rather than a fixed root.)

## Step 1 — Make `@/generated/{auth,admin}` resolve (alias onto the existing SDK)

Every data block imports its hooks from `@/generated/<namespace>`. The template generates those SDKs at
`src/graphql/sdk/<namespace>` (not `src/generated/<namespace>`). **Do not run a second codegen into
`src/generated`** — instead **alias `@/generated/{auth,admin}` onto the SDK codegen already wrote.** One
`tsconfig.json` edit, no duplicate SDK, and the template's existing `@sdk/*` aliases keep working.

Edit `<app>/tsconfig.json` (in the app dir = `packages/app`) — add two keys to `compilerOptions.paths`:

```jsonc
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"],
      "@sdk/admin": ["./src/graphql/sdk/admin"],
      "@sdk/auth": ["./src/graphql/sdk/auth"],
      "@sdk/app": ["./src/graphql/sdk/app"],

      // Blocks binding — alias @/generated/* onto the SDK codegen already wrote.
      // No second codegen; this is the preferred route (route "a").
      "@/generated/auth": ["./src/graphql/sdk/auth"],
      "@/generated/admin": ["./src/graphql/sdk/admin"]
    }
  }
}
```

> **Why aliases and not a wildcard?** `@/generated/*` → `./src/graphql/sdk/*` would also work and is what
> `check-sdk.mjs` resolves. Either is fine. **But you must NOT add a bare `@/generated/*` wildcard while a
> `@/*` → `./src/*` wildcard also exists if it would shadow real files** — keep the two explicit keys above
> unless you are certain. Next.js reads `tsconfig.json` paths directly, so no separate bundler config edit
> is required for this template.

After editing, the alias must resolve to a directory that is **non-empty and actually exports the auth
hooks blocks call** — not merely a directory that exists. An empty (or hook-less) `auth` SDK is the
real failure mode: the per-DB `auth` endpoint can be **schema-empty** (no procedures wired yet), so
codegen writes the folder but emits no `useSignInMutation`, and `test -d` passes while every block
no-ops. Verify the *content*, not just the path:

```bash
# Run in <app> (= packages/app, where the app's src/ + configs live).
# Both dirs must exist AND be non-empty…
[ -n "$(ls -A src/graphql/sdk/auth 2>/dev/null)" ] && [ -n "$(ls -A src/graphql/sdk/admin 2>/dev/null)" ] \
  || { echo "MISSING/EMPTY: re-run 'pnpm codegen' (Phase 3 / public 2.6 step 4)"; }
# …and the auth SDK must export the sign-in hook blocks depend on.
grep -rq 'useSignInMutation' src/graphql/sdk/auth \
  && echo "OK: auth SDK exports useSignInMutation" \
  || echo "AUTH SDK SCHEMA-EMPTY: no useSignInMutation — see fallback below"
```

> **Fallback — per-DB auth endpoint is schema-empty.** If the grep fails, the per-DB `auth-<subdomain>`
> endpoint has no auth procedures wired (codegen produced an empty/hook-less `auth` SDK). Re-run codegen
> against a **schema-bearing** auth endpoint so the hooks exist:
> - First confirm whether the per-DB endpoint really is empty: `curl`-introspect it (or open it in
>   GraphiQL) and look for a `signIn` mutation. If it is genuinely empty, that is a backend gap
>   (Phase 1/2 provisioning) — fix it there, do **not** hand-write hooks (CODEGEN-001).
> - If a schema-bearing auth endpoint exists for this DB (e.g. the platform `auth.localhost:3000` during
>   bring-up, or the correct `auth-<subdomain>` once provisioned), point the template's codegen config at
>   it and re-run `pnpm codegen` so `src/graphql/sdk/auth` is regenerated **in place** with
>   `useSignInMutation`. Never create a second SDK under `src/generated` (Step 1's whole point).
> - Re-run the grep above; it must print `OK` before continuing. A schema-empty auth SDK will pass
>   `check-sdk.mjs`'s directory check (Step 6) but fail the manifest-op assertion (`signIn` →
>   `useSignInMutation`), so fixing it here saves a wasted build.

## Step 2 — Host dependencies + build config

Blocks are built on `@constructive-io/ui` (consumed as an npm dependency, **never vendored**) and some
blocks (passkeys) dynamic-import `@simplewebauthn/browser`, which is **not** in any block's registry
dependencies — the host must add it.

**These are already DECLARED + INSTALLED — do not `pnpm add` them here.** `node scripts/wire-app.mjs`
(Phase 3 / S3) declared both `@constructive-io/ui` and `@simplewebauthn/browser` in `<app>/package.json`
`dependencies` (alongside the dev `@constructive-io/graphql-codegen`), and the **single** Phase-3 `pnpm
install` (SKILL.md S4) materialized them in ONE resolve of the heavy `@constructive-io/*` + graphile
tree. Running a separate `pnpm add` here re-resolves that whole tree again — the warm-time sink this
consolidation removes. Just **verify** they're present, then move on to the config edits:

```bash
# Run in <app> (= packages/app, where the app's src/ + configs live). VERIFY (do NOT re-add):
ls node_modules/@constructive-io/ui >/dev/null 2>&1 \
  || ls ../../node_modules/@constructive-io/ui >/dev/null 2>&1 \
  && echo 'OK: @constructive-io/ui installed (declared by wire-app, materialized by the S4 pnpm install)' \
  || echo 'MISSING: re-run the single `pnpm install` at the workspace root (wire-app declared the dep; do NOT pnpm add)'
```

**2a-pre. Confirm the single-`graphql` pnpm override is already in place (set in Phase 3 — F14).**
The `graphql` override is a **codegen prerequisite** and is now set back in **Phase 3** (SKILL.md
step 4a), because the generated SDK already needs a single `graphql` to build. `@constructive-io/ui`
adds nothing new here — it transitively drags graphile/postgraphile packages that also depend on
`graphql`, so without one pinned version you hit the classic *"Cannot use GraphQLObjectType … from
another module or realm"* (dual-`graphql`) error. Just **verify the Phase 3 override still holds**:

```bash
pnpm why graphql   # must list exactly ONE resolved graphql; if not, fix the Phase 3 override
```

If for some reason it is missing, add it now (workspace-root `package.json`) and re-install — but it
should already exist from Phase 3:

```jsonc
{ "pnpm": { "overrides": { "graphql": "^16.9.0" } } }   // match the version the generated SDK uses
```

**2a. Transpile the UI package** — `@constructive-io/ui` ships source that Next must transpile. Edit
`<app>/next.config.ts` (in the app dir = `packages/app`):

```ts
const nextConfig: NextConfig = {
  typedRoutes: true,
  output: 'standalone',
  transpilePackages: ['@constructive-io/ui'],   // ← add
  experimental: {
    optimizePackageImports: ['lucide-react', '@remixicon/react'],
  },
};
```

> ℹ️ **`output: 'standalone'` + `next start`:** the template ships `output: 'standalone'` (for the Docker
> runtime). `next start` works under it but prints a benign *"output: 'standalone' … next start"* warning —
> it is **not** a misconfig; for local verification prefer `pnpm dev --port 3081` (the live-QA gate also
> reuses an already-running dev server). The `GET /__runtime-config.js 404` in `next dev` is likewise
> expected (that file is only emitted in the standalone production build).

> 🚨 **`next build` type-checks the WHOLE workspace, not just the app's own `src/` (110–150s prior
> trial-and-error sink).** The workspace-root `pnpm-workspace.yaml` (`packages/*`) makes Next's bundled TS
> check pull in the sibling `packages/provision`, whose `pg`/`dotenv`/`@constructive-io/*` runtime deps
> aren't hoisted to the app's resolution context — so the build fails with *"Cannot find module"* **even
> though the app itself compiled clean**. A `tsconfig` `exclude` of `packages/provision` does **not** work
> (Next globs from the workspace root); `turbopack.root` breaks `next` package resolution. **Working fix (do
> this up front, don't debug it live):**
> 1. Set `typescript: { ignoreBuildErrors: true }` in `<app>/next.config.ts` (stops Next type-checking the sibling tree).
> 2. Gate the REAL app TS separately with a scoped `<app>/tsconfig.appcheck.json` that includes only `src/**`,
>    and run `pnpm exec tsc -p tsconfig.appcheck.json --noEmit` (in `<app>`) as your app-TS gate.
> 3. **Do NOT delete `pnpm-workspace.yaml`** — its `packages/*` glob registers BOTH `packages/app` (the app)
>    and `packages/provision`; removing it breaks the workspace. See SKILL.md speedrun S4(b)/(c).

**2b. Tailwind v4 `@source` + `@import`** — edit `<app>/src/app/globals.css` (in the app dir). Add the UI globals
import near the top (after `@import 'tailwindcss';`) and a `@source` that scans the **installed** UI
component source:

```css
@import 'tailwindcss';
@import 'tw-animate-css';
@import '@constructive-io/ui/globals.css';   /* ← brand tokens + base UI styles */

/* Existing: scan this app's own source */
@source "../";

/* ← REQUIRED. Scan the UI library's component source under node_modules.
 * @constructive-io/ui/globals.css declares its own `@source`, but Tailwind v4
 * resolves an @source inside an @import-ed sheet RELATIVE TO THAT SHEET — so the
 * UI components are NOT scanned from here. Without this line, the utilities that
 * live only inside UI components (Dialog/DropdownMenu/Popover centering, backdrops,
 * enter/leave animations) are never generated, and every modal/menu in a block
 * renders unstyled (popup not centered, no backdrop). */
@source "../../node_modules/@constructive-io/ui/dist";
```

> **The `@source` path is the trap (see gotchas BLOCKS-004).** It must point at the **installed** UI
> package's component source, RESOLVED RELATIVE TO `globals.css` (at `<app>/src/app/`). `../../` from there
> lands on `<app>/` — so `../../node_modules/@constructive-io/ui/dist` works **iff** the dep is installed in
> the app's OWN `node_modules`. But in the nested `packages/app` workspace, pnpm usually **hoists** the dep
> to the WORKSPACE-ROOT `node_modules` (one more `../` up: `../../../node_modules/…`), or the real files sit
> under `node_modules/.pnpm/@constructive-io+ui@*/…`. **Do not assume the depth — verify it (from `<app>`):**
> ```bash
> # from <app> = packages/app:
> ls node_modules/@constructive-io/ui/dist >/dev/null 2>&1 \
>   && echo 'use: @source "../../node_modules/@constructive-io/ui/dist"' \
>   || find .. ../.. -path '*/node_modules/@constructive-io/ui/dist' -maxdepth 5 -print -quit
> ```
> Use whichever path actually exists (count the `../` to reach the `node_modules` that holds it); the
> `@source` must resolve to a directory of `.js`/`.tsx` component files, or modals will render unstyled even
> though the build passes.

## Step 3 — Environment variables (the GRAPHQL-named ones)

`blocks-runtime.tsx` reads **`NEXT_PUBLIC_AUTH_GRAPHQL_ENDPOINT`** and
**`NEXT_PUBLIC_ADMIN_GRAPHQL_ENDPOINT`** — referenced as literal `process.env.*` so Next can inline them.
These are **different names** from the template's optional `NEXT_PUBLIC_{AUTH,ADMIN}_ENDPOINT` overrides.
Set the GRAPHQL-named vars (this is gotchas BLOCKS-001):

Append to `<app>/.env.local` (in the app dir; copy `<app>/.env.example` → `<app>/.env.local` first; keep
the existing `NEXT_PUBLIC_DB_NAME=<db>` line — or just run `node scripts/wire-app.mjs --app <app> --sub
<sub>`, which writes all of these). Here
`<sub>` is the **subdomain, which == the database name with underscores** (db `my_app` → `my_app`);
discover the real per-DB endpoints from `services_public.domains` (Phase 2.3, SUBDOMAIN-001) — do
**not** assume `app-public-`:

```bash
# Blocks runtime reads these EXACT names (note the _GRAPHQL_ infix).
NEXT_PUBLIC_AUTH_GRAPHQL_ENDPOINT=http://auth-<sub>.localhost:3000/graphql
NEXT_PUBLIC_ADMIN_GRAPHQL_ENDPOINT=http://admin-<sub>.localhost:3000/graphql
```

> If you only set `NEXT_PUBLIC_AUTH_ENDPOINT` / `NEXT_PUBLIC_ADMIN_ENDPOINT`, blocks will warn
> `Missing NEXT_PUBLIC_AUTH_GRAPHQL_ENDPOINT` at runtime and every block request will no-op. Set the
> `_GRAPHQL_` names. (The template's own SDK keeps deriving endpoints from `NEXT_PUBLIC_DB_NAME` — leave
> that intact; these new vars are additive, only for blocks-runtime.)

> 🚨 **The app DATA endpoint is `api-<sub>`, NOT `app-public-<sub>` (F2 — see
> VERIFICATION-FINDINGS.md).** Your business-entity CRUD (the `app` namespace) routes on
> **`http://api-<sub>.localhost:3000/graphql`**. Two non-obvious traps:
> - **The HTTP `Host` header drives schema routing — a URL override alone still 404s.** The per-DB
>   schema is selected from the request's `Host`, so the `Host` must be `api-<sub>.localhost` (not just
>   the URL). The template hardcodes this; make both layers env-overridable:
>   - **codegen** — honor a `CODEGEN_APP_HOST` env var for the introspection Host
>     (`graphql-codegen.config.ts`), so codegen hits `api-<sub>` with the matching `Host`.
>   - **runtime** — honor `NEXT_PUBLIC_APP_ENDPOINT` / a `getAppEndpoint()` override
>     (`config-core.ts`) so the running app's `app` adapter targets `api-<sub>` too.
> - **Discover, don't assume.** Read the per-DB domains from `services_public.domains`; `<sub>` is the
>   db name with underscores. To confirm before trusting it:
>   `curl -s http://localhost:3000/graphql -H 'Host: api-<sub>.localhost' -H 'content-type: application/json' -d '{"query":"{ __typename }"}'`
>   (then check your table appears in the schema). This is the same `api-<sub>` correction the SKILL.md
>   Phase 3 endpoint list and SUBDOMAIN-001 carry.

## Step 4.0 — Pick your flow(s) first (the catalog drives everything below)

**Before you install anything, pick the backend-capability _flow(s)_ this app needs from
`references/flows.json`.** A "flow" is one auth/account/org capability bundle (the same model
better-auth uses for its plugins) and it is the **single source of truth** that ties together (a) which
backend modules Phase 2 had to provision and (b) the exact set of Blocks to install here. `flows.json`
is generated from the apps/blocks manifest and is **machine-checked** (`pnpm check:flows`) — do **not**
hand-edit it, and do not guess block names or module lists when a flow already spells them out.

```bash
# List the available flow ids + their group, from the generated catalog (no jq needed):
node -e 'const f=require("./references/flows.json");for(const x of f.flows)console.log(x.group.padEnd(16),x.id,"—",x.name)'
```

Map the brief's auth needs to flow id(s). Representative picks (see `flows.json` for the full ~18):

| The brief asks for… | flow id(s) | group |
|---|---|---|
| email + password sign-up / sign-in / sign-out | `email-password` | authentication |
| verify-email link + resend banner | `email-verification` | authentication |
| forgot / reset password | `password-reset` | authentication |
| “sign in with Google/GitHub” | `social-oauth` | authentication |
| account settings / profile editing | `profile` | account-session |
| change password, manage sessions, API keys, delete account | `change-password`, `sessions`, `api-keys`, `account-deletion` | account-session |
| organizations / teams / workspaces (+ members, roles, invites) | `organization`, `org-members`, `org-roles`, `org-invites`, `app-memberships` | authorization |

Once you have your flow id(s), read those entries in `flows.json`. **Two fields are load-bearing:**

1. **`flow.blocks`** — the **exact set of blocks to install** in Step 4 below, by their **real prefixed
   names** (e.g. `email-password` → `auth-sign-in-card`, `auth-sign-up-card`, `auth-sign-out-button`,
   `auth-sign-in-page`, `auth-sign-up-page`). This is the authoritative answer to "which blocks?" — you do
   **not** need to guess from the `ls` in Step 4a (use it only to confirm the names resolve). Installing a
   flow = installing exactly its `blocks` list.
2. **`flow.backend.modules`** (with `flow.backend.preset` as the advisory anchor) — the **exact module set
   Phase 2 must already have provisioned** for this flow's ops to exist. This is the same list PROVISION-001
   talks about: e.g. every authentication / account-session flow rides `auth:email`; `social-oauth` /
   `connected-accounts` need `auth:sso` (adds `connected_accounts_module` + `identity_providers_module`);
   every `org-*` flow needs `b2b`. **`modules` is authoritative; `preset` is only the smallest covering
   shipped preset** (org flows have no preset smaller than `b2b`).

**B2B detection (the opt-in tier).** A flow is b2b iff its `backend.modules` include the **org-scoped**
`memberships`/`hierarchy` tuples (`['memberships_module',{scope:'org'}]`, `['hierarchy_module',{scope:'org'}]`)
— equivalently `backend.preset === 'b2b'`. For a b2b flow you do **two** things: provision the org modules
(Phase 2 — PROVISION-001) and install that flow's **org blocks** here in Step 4 (the `org-*` flows resolve to
`org-create-card`, `org-members-list`, `org-roles-editor`, `org-settings-form` + the invite/membership blocks
— read each flow's `blocks` array). A base `auth:email` flow has none of these tuples, installs none of the
org blocks, and needs no org provisioning.

**Reconcile against Phase 2 before installing.** If your chosen flow's `backend.preset`/`modules` is richer
than what Phase 2 provisioned (e.g. you picked `social-oauth` (auth:sso) or any `org-*` flow (b2b) but the
DB was provisioned with the `auth:email` list), the blocks will install but their ops won't exist and
`check-sdk.mjs` (Step 6) will fail on the missing op. The fix is to **re-provision with the flow's module
list** (PROVISION-001) — not to hand-write hooks. The `flow.backend.exposedOps` array is exactly what
`check-sdk.mjs` will look for, so a flow whose ops are absent is a provisioning gap surfaced honestly.

> A flow's `blocks` list and its `backend.modules` list are kept in lockstep by the generator + `pnpm
> check:flows`; that is the whole point of the catalog. Treat the flow id as the unit of work: "install the
> `email-password` flow" means *provision its modules (Phase 2) + install its blocks (Step 4) + wire them
> (Step 5)*, all read off one `flows.json` entry.

## Step 4 — Install a block

> You already chose the flow(s) in Step 4.0; the names you install below are exactly the entries in each
> chosen flow's `blocks` array. Step 4a is now just a sanity check that those names resolve in the built
> registry.

### 4a. Discover the REAL block names first (they are prefixed)

Registry block names are **namespace-prefixed** — they are **not** the short names you might guess.
There is no `auth-sign-out`; the real entry is **`auth-sign-out-button`**. There is no `account-profile`;
it is **`auth-account-profile-card`**. **List the real names from the registry before you `add`
anything** — guessing a name yields a 404/"not found", which looks like a registry outage but is just a
wrong name:

```bash
# List every available block name from the live @constructive registry (GitHub Pages, see 4c).
# The index lists every published block; FLOW blocks (auth-*/org-*) live here alongside the UI primitives:
curl -sf https://constructive-io.github.io/dashboard/r/index.json \
  | node -e 'let s="";process.stdin.on("data",d=>s+=d).on("end",()=>{for(const e of JSON.parse(s))console.log(e.name)})' \
  | sort
# (Fallback, local serve — see 4c: ls .registry-cache/r/*.json or the path serve-registry.sh resolves.)
```

Representative **real** names (prefixed): `auth-sign-in-card`, `auth-sign-up-card`,
`auth-sign-out-button`, `auth-forgot-password-card`, `auth-reset-password-card`,
`auth-account-profile-card`, `auth-account-security-card`, `auth-account-sessions-list`,
`auth-mfa-totp-enroll`, `auth-passkey-sign-in`, `auth-invitation-acceptance-card`,
`org-create-card`, `org-members-list`, `org-invite-dialog`, `org-settings-form`, `user-avatar`.
Use the exact name from the registry listing above — pick the one whose name matches the flow you need.

### 4b. Install by NAME against the registry — the raw local-file `add` 404s, do NOT use it

A naïve `npx shadcn add /abs/path/.../auth-sign-in-card.json` (or a bare `add @constructive/<block>` with a
mis-set registry) **fails with a 404.** The block JSON lists `@constructive/*` **registryDependencies** (e.g.
`blocks-runtime`, `cn`, UI primitives), and shadcn resolves each of those against the registry URL in
`components.json` → `registries.@constructive`. Point that key at the **live published `@constructive`
registry** (`https://constructive-io.github.io/dashboard/r/{name}.json`) and install by name — the
transitive deps resolve there. The published registry is the **default and preferred** path (4c); the
local HTTP serve is only a fallback when you are off-network or pinning an unpublished block.

### 4c. Working path — point `components.json` at the published registry, then `add` (local serve = fallback)

### 4d. Delete the stray `src/lib/utils.ts` the `cn` dep writes — it silently shadows the template's `src/lib/utils/` dir (gotchas BLOCKS-007)

Blocks depend on a `cn` registry item that writes a **file** `src/lib/utils.ts` (its registry file
resolves to `@/lib/utils`). But the template does **not** have a `src/lib/utils.ts` file — it ships a
**`src/lib/utils/` directory** whose barrel `index.ts` re-exports `common.utils` (exports `cn`, `clamp`)
and `file.utils` (`getImageUrl`, …), and **57 files import `@/lib/utils`**. Because `tsconfig.json`
maps `@/*` → `./src/*`, the new **file** `src/lib/utils.ts` resolves *before* the directory index
`src/lib/utils/index.ts` (file beats dir-index), so it **silently shadows the whole barrel** — dropping
`file.utils` (and any `common.utils` export beyond `cn`) — and breaks those 57 importers.

There is **no overwrite prompt** (the file and the directory's files are different paths, so `shadcn add`
writes it additively without asking). After adding any block, check for the stray file and **delete it**:

```bash
# Run in <app>. The template ships src/lib/utils/ (directory).
# If shadcn's cn dep wrote a sibling FILE, remove it:
ls -la src/lib/utils.ts 2>/dev/null && rm src/lib/utils.ts
# Confirm @/lib/utils resolves back to the barrel and still exports cn + the rest:
test -f src/lib/utils/index.ts && echo "barrel intact"
```

`rm` (not `git checkout`) is the fix: the `src/lib/utils/` directory and its files are git-tracked, but
the stray `src/lib/utils.ts` is **not** tracked, so `git checkout -- src/lib/utils.ts` would do nothing.
The template's `cn` is compatible, so deleting the stray file costs no block functionality. See gotchas
BLOCKS-007.

## Step 5 — Provider wiring (mount BlocksRuntime + StepUpProvider; resolve the collision)

`BlocksRuntime` is the **one** wiring point: it configures each namespace's `configure()`, reuses the
host's `QueryClient`, and attaches `Authorization: Bearer <token>` per request via a host-supplied
`getToken`. Some account/MFA/passkey blocks also require **`StepUpProvider`** at the root (gotchas
BLOCKS-006). Keep the template's `AppPortalRoot` exactly as is.

> 🚨 **The auth blocks do NOT persist the token themselves — you must bridge `onSuccess`.** `SignInCard` /
> `SignUpCard` fire an `onSuccess` callback with `{ accessToken, userId }` and then stop; they never write to
> the template's `TokenManager`/session. So `getToken={() => TokenManager.getToken('admin').token?.accessToken}`
> keeps returning `undefined` after a "successful" sign-in, and every subsequent request goes out **anonymous**
> — the user looks signed in but all data calls 401/return 0 rows. **Wire the host bridge:** pass an
> `onSuccess` to each auth card that stores the returned token into the template's `TokenManager` (the same
> session store `getToken` reads). The template setter is `TokenManager.setToken(token: ApiToken, rememberMe,
> 'admin')` where `ApiToken = { userId, accessToken, accessTokenExpiresAt }` (see
> `src/lib/auth/token-manager.ts` + `src/store/auth-slice.ts`), e.g. `onSuccess={({ accessToken, userId,
> accessTokenExpiresAt }) => TokenManager.setToken({ userId, accessToken, accessTokenExpiresAt }, true,
> 'admin')}` — map the block's `onSuccess` payload onto the template's real `ApiToken` shape. Without this
> `onSuccess → TokenManager` bridge the token is dropped on the floor; this is a recurring integration stumble
> distinct from the stale-anonymous-header bug in 5a (there the token exists but the `app` adapter snapshotted
> headers before it arrived).

> 🚨 **`SignUpCard` defaults to a 3-field form (`showPasswordConfirm` + `showPasswordStrength` both
> default `true`) — but the `email-password` live-QA contract submits only `email` + `password`.** With the
> confirm field shown, a 2-field submit is **blocked** (the confirm is empty/mismatched) and the live-QA
> gate's signup step hangs/fails on the un-clicked submit. For the `email-password` flow, mount the card (or
> the page that wraps it — `auth-sign-up-page`) with **`<SignUpCard showPasswordConfirm={false}
> showPasswordStrength={false} onSuccess={…}/>`** so the scaffolded `/sign-up` matches the driver's two-field
> happy path (`scripts/live-qa.mjs` fills `email` + `password`, then clicks `sign-up-submit`). If you keep the
> confirm field for a real UX, the live-QA driver must also fill it (set `LIVE_QA_*` testids) — but the
> two-field form is the documented default for the golden/email-password flow.

**5a. Resolve the double-`configure()` collision (REQUIRED — gotchas BLOCKS-002).** The template's
`AppProvider` calls `configureAuth` + `configureAdmin` + `configureApp` at module load. `BlocksRuntime`
**also** configures `auth` + `admin`. Two configurers per namespace is last-writer-wins and will silently
send block requests through the wrong adapter (no bearer token). Fix it so there is **one configurer per
namespace**: remove `configureAuth` and `configureAdmin` from `AppProvider`, **keep `configureApp`** (the
template still owns the `app` namespace; BlocksRuntime owns `auth` + `admin`).

> ✅ **AUTO-APPLIED by `wire-app.mjs` (GAP-A fix) — you no longer wire this by hand.** The retained
> `configureApp` runs at MODULE LOAD (before login) and the generated SDK's `FetchAdapter` SNAPSHOTS
> `config.headers` ONCE at construction (`new FetchAdapter(endpoint, config.headers, …)`), so even the
> template's `get headers()` getter is read exactly once → it snapshots an **EMPTY** header set and the
> `app` namespace stays ANONYMOUS after sign-in (gotchas SDK-008). Unlike `BlocksRuntime` (which attaches
> the token *per request* via `getToken`), the snapshotted `app` adapter has no bearer, so every `@sdk/app`
> CRUD op runs as the anonymous role → `permission denied` returned as **HTTP 200 with 0 rows**, and the
> very FIRST create/list in a fresh authenticated session (right after sign-up/sign-in, **before any full
> page reload**) silently fails. A reload only "fixes" it because the module re-runs with the token already
> in storage. **A 200-but-0-rows on your OWN `app_public` table is almost always this stale-anonymous-header
> bug, NOT an RLS policy gap — check the request's `Authorization` header before suspecting RLS.**
>
> **`node scripts/wire-app.mjs` now injects a PER-REQUEST app-token seam into `app-provider.tsx`
> automatically** (step c2): it adds a `createAuthedFetch(ctx)` helper that wraps the SDK runtime's
> `createFetch()` and merges the **live** `getAuthHeaders(ctx)` (→ `TokenManager`) on **every** request,
> then passes it as the SDK config's `fetch`. Because the token is re-read per request it is never stale
> across sign-up / sign-in / refresh / logout, and the first create works with no reload. Verified live:
> an anonymous create returns `permission denied for table <t>`; the same create with the fresh bearer
> succeeds + persists.
>
> The two equivalent **manual** fixes (only needed if the template shape drifted and wire-app printed a
> WARN that it could not inject the seam) are:
> - **Wire `app` to read the token per request (preferred — what wire-app does):** give the `app` SDK config
>   a custom `fetch` (wrapping `createFetch()`) that merges `getAuthHeaders('app')` on every call — equivalently
>   `() => TokenManager.getToken('app').token?.accessToken` for an adapter-style template — so each request
>   picks up the current token and never goes stale; **or**
> - **Re-configure `app` after login:** call `configureApp({ adapter: createSdkAdapter('app') })` again in
>   your auth-success callback / a post-login session effect, so the adapter is rebuilt with the live token.

Edit `<app>/src/components/app-provider.tsx` — **`wire-app.mjs` does BOTH of these for you**: (1) the
collision removal below, AND (2) the per-request app-token seam from the GAP-A callout above (it injects a
`createAuthedFetch(ctx)` helper + a `fetch:` property on the `app` SDK config):

```diff
- import { configure as configureAdmin, type QueryResult } from '@sdk/admin';
- import { configure as configureAuth } from '@sdk/auth';
+ import { type QueryResult } from '@sdk/admin';
  import { configure as configureApp } from '@sdk/app';
  ...
- // Initialize all SDK clients at module load time (before any queries can fire).
- configureAdmin({ adapter: createSdkAdapter('admin') });
- configureAuth({ adapter: createSdkAdapter('auth') });
- configureApp({ adapter: createSdkAdapter('app') });
+ // BlocksRuntime now owns auth + admin configuration (one configurer per namespace).
+ // AppProvider keeps configuring only the `app` namespace (the template's business SDK).
+ configureApp({ adapter: createSdkAdapter('app') });
```

The getter-style template variant (`createSdkConfig(ctx) → { get endpoint(){…}, get headers(){…} }`) gets the
seam added as a sibling property — wire-app injects `fetch: createAuthedFetch(ctx),` so the bearer is attached
per request even though `get headers()` is only read once at construction:

```diff
+ import { createFetch } from '@constructive-io/graphql-query/runtime';
  ...
+ function createAuthedFetch(ctx: SchemaContext): typeof globalThis.fetch {
+   const baseFetch = createFetch();
+   return ((input, init) => {
+     const merged = new Headers((init && init.headers) || {});
+     merged.delete('authorization');
+     for (const [k, v] of Object.entries(getAuthHeaders(ctx))) merged.set(k, v);
+     return baseFetch(input as any, { ...(init || {}), headers: merged });
+   }) as typeof globalThis.fetch;
+ }
  function createSdkConfig(ctx: SchemaContext) {
    return {
      get endpoint() { return getEndpoint(ctx); },
      get headers() { return getAuthHeaders(ctx); },
+     fetch: createAuthedFetch(ctx),
    };
  }
```

> If `QueryResult` becomes the only remaining `@sdk/admin` import and your linter flags an unused
> `createSdkAdapter('admin')`/`('auth')` path, that is expected — `createSdkAdapter` is still used for
> `'app'`. Leave the factory; just drop the two `configure*` calls and their imports.

**5b. Mount `BlocksRuntime` + `StepUpProvider` via a `'use client'` wrapper — NOT directly in the server
`layout.tsx`.** `BlocksRuntime` is a **Client Component** (`'use client'`) and takes a **function** prop,
`getToken: () => string | null | undefined | Promise<…>`. The sandbox template's `src/app/layout.tsx` is
a **Server Component**. If you mount `<BlocksRuntime getToken={() => …}>` *directly* in that server
layout, Next throws a build error:

> Functions cannot be passed directly to Client Components unless you explicitly expose it by marking it
> with `"use client"`. ... `getToken={function getToken}`

because a server file cannot hand a closure across the server→client boundary. **Fix: put the runtime in
a small `'use client'` Providers wrapper** and import that wrapper into the server layout (the wrapper
owns the function, so no function crosses the boundary).

Create `<app>/src/components/blocks-providers.tsx` (this is what `wire-app.mjs` writes for you):

```tsx
'use client';

import type { ReactNode } from 'react';
import { BlocksRuntime } from '@/blocks/runtime/blocks-runtime';
import { StepUpProvider } from '@/blocks/auth/use-step-up/step-up-provider';
import { TokenManager } from '@/lib/auth/token-manager';

// Owns the getToken closure on the client side, so layout.tsx (a Server Component)
// never passes a function across the server→client boundary.
export function BlocksProviders({ children }: { children: ReactNode }) {
  return (
    <BlocksRuntime
      namespaces={['auth', 'admin']}
      getToken={() => TokenManager.getToken('admin').token?.accessToken}
    >
      <StepUpProvider>{children}</StepUpProvider>
    </BlocksRuntime>
  );
}
```

`BlocksRuntime` detects the host's existing `QueryClientProvider` and reuses it, so mount the wrapper
**inside** `AppProvider` (below the existing `QueryClientProvider`). Edit
`<app>/src/app/layout.tsx` (note: only a component import crosses in — no function; `wire-app.mjs` does this):

```diff
  import { AppProvider } from '@/components/app-provider';
+ import { BlocksProviders } from '@/components/blocks-providers';
  ...
            <AppProvider>
+             <BlocksProviders>
                <CardStackProvider layoutMode='side-by-side' defaultPeekOffset={48}>
                  {/* …existing children unchanged… */}
                </CardStackProvider>
+             </BlocksProviders>
            </AppProvider>
```

(`AppPortalRoot` stays where it is, at the end of `<body>` — blocks' dialogs/menus portal into it.) The
exact import paths for `BlocksRuntime` / `StepUpProvider` are wherever the registry installed them; the
paths above match the dashboard layout (`src/blocks/runtime/...`, `src/blocks/auth/use-step-up/...`).
Confirm with `ls src/blocks` after install. If the template's `layout.tsx` already carries `'use client'`
you could inline the runtime, but the wrapper is the safe default and keeps the server layout a Server
Component.

> **Token shape:** `TokenManager.getToken('admin')` returns `{ token: ApiToken | null, rememberMe }`;
> the JWT for the `Authorization: Bearer` header is `token.accessToken`. Passing
> `() => TokenManager.getToken('admin').token?.accessToken` gives BlocksRuntime a fresh token per request
> (it never goes stale across login/refresh/logout).

## Step 6 — Preflight with `check-sdk.mjs`

Before you build, prove the host SDK actually exports every operation each installed block needs. Run the
skill's bundled preflight with `--project <app>` (so it reads `<app>/tsconfig.json` + the manifests under
`<app>/src/.constructive/blocks`):

```bash
# check-sdk.mjs is bundled in THIS skill's scripts/ — no cross-repo path. Run from the skill root:
node scripts/check-sdk.mjs --project <app>
```

It (1) verifies the `@/generated/*` alias exists in `tsconfig.json`, (2) resolves the generated dir for
each block's namespace and confirms it exists, (3) asserts every manifest op maps to a real SDK export
(`signIn` → `useSignInMutation`), and (4) advises whether `<BlocksRuntime>` is mounted. **Exit 0** =
satisfied; **exit 1** = a prerequisite missing (it prints the exact remediation); **exit 2** = could not
run (no tsconfig / bad manifest).

> **One adaptation for this template:** on a red check, `check-sdk.mjs` prints a generic remediation that
> ends in `… --orm -o src/generated` (its default assumes the SDK lives at `src/generated`). For **this
> template do NOT run that** — it would create the very second SDK Step 1 tells you to avoid. The
> template's SDK already lives at `src/graphql/sdk/{auth,admin}` and is aliased there. Regenerate **in
> place** with the template's own command (`pnpm codegen`, Phase 2.6 step 4), then re-run the preflight.
> Only the *namespace/op* the check names is signal; ignore its `-o src/generated` path.

A red check naming a missing op means the **backend has not deployed that procedure** (or the SDK is
stale) — that is the gap surfacing honestly. Do not hand-edit generated files or guess; regenerate the
SDK in place (Phase 2.6 step 4 → `pnpm codegen`, which writes `src/graphql/sdk/*`) or treat the op as
backend-pending per the skill's "Known SDK gaps" table.
